## Content Architecture

### Content Collections

Two content collections defined in `src/content.config.ts`:

1. **`docs`** — Standard Starlight documentation (guides, reference, tutorials).
   - Path: `src/content/docs/`
   - Format: `.md` and `.mdx`
   - Has sidebar, navigation, and search support.

2. **`synadrive`** — Game design documents auto-synced from Obsidian vault.
   - Path: `src/content/synadrive/` (also `synadrive-web/src/content/synadrive/`)
   - Source of truth: Obsidian vault at `Life/Gamedev/Synadrive/`
   - Synced via GitHub Actions workflow (`sync_to_web.yml`).
   - **Do NOT manually edit** files in the `synadrive` collection — they will be overwritten on next sync.

### Documentation Strategy (Hybrid)

| Type | Content | Location |
|---|---|---|
| **API Reference** | Auto-generated from code comments + hand-curated summaries. | `docs/reference/` |
| **Guides** | Step-by-step tutorials for using game systems. | `docs/guides/` |
| **Game Design** | High-level concepts, mechanics, vision docs. | `synadrive/` (from Obsidian) |
| **Developer Docs** | Contributing, architecture, debugging. | `docs/developer/` |

## Writing Conventions

### Frontmatter (Starlight docs)
```yaml
---
title: Vehicle Physics System
description: How the Pacejka tire model and suspension simulation work in Synadrive.
sidebar:
  order: 1
  badge:
    text: WIP
    variant: caution
---
```

### MDX Components
- Use Starlight's built-in components: `<Aside>`, `<Tabs>`, `<Card>`, `<CardGrid>`, `<Steps>`.
- Import custom components from `src/components/` when needed.
- Keep MDX component usage minimal — prefer standard Markdown for portability.

### Code Blocks
- Always specify language: ` ```gdscript `, ` ```cpp `, ` ```typescript `.
- Add titles for context: ` ```gdscript title="vehicle_controller.gd" `.
- Use line highlighting for focus: ` ```gdscript {3-5} `.

### Style
- Write in **second person** ("you") for guides.
- Write in **third person** for reference docs.
- Keep paragraphs short (3-4 sentences max).
- Use tables for structured data (properties, parameters, signals).
- Include diagrams (Mermaid) for architecture and data flow.

## Obsidian Sync Awareness

The `synadrive` content collection is **automatically synced** from the Obsidian vault:
- **Source:** `Life/Gamedev/Synadrive/` in the Obsidian repo.
- **Destination:** `docs/synadrive-web/src/content/synadrive/` (workspace-relative from the game `src/` root). Locally this may appear as the double-nested path `synadrive-web/synadrive-web/src/content/synadrive/` in VS Code — this is a workspace folder naming artifact, not an error.
- **Workflow:** `sync_to_web.yml` in the **Obsidian vault’s** `.github/workflows/` directory (NOT in the game repo’s `.github/workflows/`). It triggers on push to `main` in the Obsidian repo `VitorMPCastro/dotcom` and pushes content to this docs site repo.

**Rules:**
- Files in `src/content/synadrive/` are **read-only** from this repo's perspective.
- To update game design docs, edit them in Obsidian — they’ll sync automatically.
- **Wikilink warning:** Obsidian `[[wikilinks]]` and `![[embeds]]` will NOT render in Astro/Starlight without a `remark-wiki-link` plugin. Either add the plugin to `astro.config.mjs` or convert wikilinks to standard markdown links during the sync workflow.
- `.base` and `.canvas` files from Obsidian are non-content metadata — ignore them.
- Dataview queries in Obsidian docs will render as raw code blocks on the web. They are Obsidian-only features.

## When to Ask Questions

Agents working on the docs site should use `ask_questions` before:
- Creating a new content collection or modifying `content.config.ts`.
- Changing the sidebar structure or navigation hierarchy.
- Adding new Astro integrations or Starlight plugins.
- Writing documentation for a game system that isn't fully implemented yet (confirm scope and accuracy).
- Modifying the Obsidian sync workflow.
- Adding custom components or layouts beyond Starlight defaults.
